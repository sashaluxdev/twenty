# Plan: Quality sweep, then ADR 0017, then ADR 0018

Three phases, strictly in sequence. Each phase ends with a user checkpoint
before the next begins. Execution model: subagent-driven development
(orchestrator dispatches; Opus for decision/nuance tasks, Sonnet for
can't-mess-up fetch/explore tasks).

## Phase 0 — Holistic quality sweep (TEST-LESS by default)

One deliberate pass over the whole app to catch what previous feature-focused
passes missed. Scope: everything under
`packages/twenty-apps/community/formula-field/` — engine, logic-functions,
front-components, objects, views, page-layouts, roles, navigation-menu-items,
README, context.md, and all ADRs.

**Hunt for three things:**

1. **Legacy/dead code** — unused exports, superseded helpers, code paths made
   unreachable by later ADRs, leftover scaffolding from abandoned directions.
2. **Comment/doc drift** — comments, README, context.md, and ADR claims that
   no longer match the code. Default remedy: fix the doc to match the code.
   If the CODE looks wrong relative to a documented invariant, that is a
   latent-bug finding, not a doc fix — escalate, don't silently "fix" either
   side.
3. **Latent bugs** — real defects that evaded prior passes. Verify each
   suspected bug against actual behavior (trace the code path end to end;
   don't pattern-match). Fix only what is confirmed.

**Known-bug hint (start here for the latent-bug track):** the record
variations feature does NOT work on the cloud deployment — opening its tab
flashes "loading" then goes blank immediately. It was live-verified working
locally (2026-07-08), so suspect a cloud-specific difference: front-component
runtime/bundling differences, a swallowed error in the tab's data fetch,
missing object/permission/seed data on cloud, or an environment assumption
(e.g. local server URL, auth context) baked into the front-component.
Relevant surfaces: `src/front-components/` variation tab entry,
`src/objects/variation-config.object.ts`, the variation logic functions, and
whatever the tab fetches on mount. Diagnose root cause before fixing;
systematic-debugging discipline applies.

**Test policy for this phase (differs from repo default — deliberate):**

- NO new tests are written during the sweep. Fixes land test-less.
- Existing tests MUST stay green after every fix
  (`cd packages/twenty-apps/community/formula-field && npx vitest run`).
- Any HIGH-VALUE test identified along the way is *designed but deferred*:
  record file, test name, scenario, setup, and assertions — enough that it
  could be implemented without re-investigation. All deferred test designs
  are presented in ONE batch at the end of Phase 0 for the user to approve,
  cherry-pick, or implement themselves. Do not implement any of them without
  explicit approval.

**Constraints:**

- Preserve every invariant documented in the ADRs (booleans out of the value
  domain, null propagation per ADR 0003, errors-always-fire, reserved-word
  dispatch, etc.). ADR 0017 lists the current verified set.
- No behavior changes without confirmed evidence of a defect.
- Small, single-concern commits (drift fixes separate from bug fixes).
- Never delete or rewrite committed instance-command logic (repo rule).

**Phase 0 exit deliverables (user checkpoint):**

1. Findings report: what was found, what was fixed, what was deliberately
   left alone (and why), cloud-variants root cause + fix status.
2. The deferred high-value test batch, awaiting approval.
